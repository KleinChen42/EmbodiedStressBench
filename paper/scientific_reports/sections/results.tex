\subsection{Protocol and experimental design}

Figure~\ref{fig:scirep-protocol} summarizes the diagnostic protocol. Each episode starts from a language query and a detector or selector, lifts the selected RGB-D evidence to a 3D target, applies a parameterized stressor, and records target error, success, oracle gap, and failure type. Table~\ref{tab:scirep-experiment-design} summarizes the evidence sets used in the manuscript; the full stressor-level definitions are provided as Supplementary Table S1. Detection-list semantic ambiguity stressors insert distractor detections before the target to test selector-order sensitivity. They should not be interpreted as physically inserted scene distractors unless a rendered scene explicitly shows one.

These gaps support the interpretation that failures are produced by the target-generation stage rather than only by task infeasibility.

\subsection{Why oracle diagnostic success can be below one}

The oracle target uses privileged simulator geometry, but it is not a mathematical guarantee of success under every diagnostic row. Oracle rows are evaluated with the same finite target-distance threshold and stressor protocol as non-oracle rows. Consequently, oracle diagnostic success can fall below one when hard stressors move the evaluated target, when projection or visibility artifacts affect the stored diagnostic target, or when the stressor intentionally audits downstream target-displacement tolerance. Table~\ref{tab:oracle-diagnostic-success} reports the observed oracle success rates and prevents the stronger but incorrect interpretation that oracle rows are forced to be perfect.

\subsection{Target sources show a precision--robustness tradeoff}

The default success threshold is 0.08 m, but the stored target-error values allow the same results to be recomputed at stricter and more relaxed thresholds. Table~\ref{tab:threshold-sensitivity} shows that crop-median depth is strongest at the default and relaxed thresholds, whereas box-center depth is more precise under strict 0.04 m and 0.06 m thresholds. We therefore avoid the stronger claim that crop-median depth is universally best. The evidence supports a precision--robustness tradeoff: box-center depth tends to place targets tightly when localization is clean, while crop-median depth better tolerates mixed-depth crops at operational thresholds.

\subsection{Open-vocabulary detector bridge exposes detector and adapter limitations}

The detector bridge evaluates learned and controlled selectors through the same target-source interface. In Bridge v2, GroundingDINO produces usable detections but has lower diagnostic success than metadata query-aware selection, and its low 2D IoU does not imply absence of all useful 3D evidence. Diagnostic success is computed from lifted 3D target distance, so a detection box can fail a strict 2D IoU criterion while still preserving enough crop depth for a valid target in simple scenes.

The detector-valid Bridge v2 rows show that GroundingDINO can be plugged into the same protocol and produces explicit detector-specific failure modes rather than requiring a separate evaluation stack (Table~\ref{tab:scirep-open-vocab-transfer}). We write these outcomes as detector-query and adapter-label limitations in the benchmark, not as a general claim that GroundingDINO cannot localize YCB or clutter objects. The result strengthens the benchmark claim because the same artifact schema captures success, no-detection, wrong-detection, and target-error outcomes across controlled and learned selectors.

\subsection{External real RGB-D validation separates detector and lifting failures}

To test whether the diagnostic logic is only a ManiSkill artifact, we added an external RGB-D probe on the full YCB-V/BOP test split with ground-truth visible masks and boxes. This probe does not evaluate robot execution. Instead, it uses the visible-mask median as a 3D reference, then compares oracle masks, oracle 2D boxes, first ground-truth boxes, and GroundingDINO detections through the same RGB-D target sources. The full probe contains 98,567 object instances and 1,577,072 target-generation records per detector-threshold setting. The oracle-mask control reaches 99.97\% success at 8 cm, showing that the reference and depth-lifting machinery are internally consistent. Oracle 2D boxes with crop-trimmed median depth reach 79.9\%, while GroundingDINO with generic prompts reaches 23.8\% and improves to 61.4\% with true object-name prompts (Table~\ref{tab:external-rgbd-validation}; Fig.~\ref{fig:external-rgbd-validation}). Thus, the external probe supports a bounded claim: EmbodiedStressBench can separate RGB-D lifting error from detector/query error on real RGB-D data, but it does not establish real-robot manipulation performance.

The query experiment therefore supports a bounded detector-query/domain and adapter-label limitation, rather than a broad detector leaderboard claim.

\subsection{Closed-loop execution calibration remains unresolved}

We attempted a scripted closed-loop oracle calibration to test whether diagnostic target success can be treated as a proxy for actual task success. The rescue gate completed without duplicate files and produced nonzero but still insufficient oracle task success for PickCube and PickSingleYCB (Table~\ref{tab:closed-loop-v2-gate}). This fails the pre-registered oracle gate. Consequently, the current manuscript does not claim that diagnostic target success predicts closed-loop task success. Instead, the result is used to define a limitation and a future calibration requirement.
