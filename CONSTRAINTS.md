# Project Constraints and Research Boundaries

## Hardware setting

The project assumes access to strong compute, such as an 8×H200 server, but no real robot arm.

## What the project can claim

The project may claim:

- simulation-based stress testing;
- reproducible benchmark design;
- systematic failure diagnosis;
- robustness analysis under controlled perturbations;
- oracle-gap analysis between privileged targets and predicted targets;
- engineering efficiency enabled by AI-assisted coding, if measured.

## What the project must not claim

The project must not claim:

- real-robot validation;
- sim-to-real success;
- universal embodied intelligence;
- state-of-the-art robot manipulation performance;
- real deployment readiness;
- results not generated from saved experiment outputs.

## Research integrity requirements

- All result numbers must be traceable to `outputs/**/*.json` or `outputs/**/*.jsonl`.
- All plots and tables must be generated by scripts.
- All experiment matrices must be specified in YAML.
- All errors should be logged as structured records.
- All random perturbations must use explicit seeds.

## Reviewer-facing framing

Use this framing:

> We propose a simulation-based diagnostic benchmark for language-conditioned manipulation pipelines. Our goal is not to solve robot manipulation, but to expose hidden fragility under semantic, visual, geometric, and execution perturbations.

Avoid this framing:

> We build a general embodied AI system that transfers to real robots.
